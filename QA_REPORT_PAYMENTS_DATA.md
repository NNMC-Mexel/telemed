# QA-отчёт: кейсы данных и оплаты (Senior QA review)

**Дата:** 2026-06-06
**Объём:** сквозной флоу записи на приём и оплаты — фронтенд (`BookingModal.jsx`, `PaymentSuccess.jsx`), платёжный шлюз (`signaling-server/server.js`), контроллер и lifecycle записей (`appointment.ts`, `lifecycles.ts`), payment-intent.
**Методология:** статический разбор кода + моделирование сценариев (happy-path, negative, race/TOCTOU, отказы сети/шлюза, idempotency, безопасность, граничные по времени/деньгам). Тесты не запускались — в проекте нет автотестов на этот флоу (см. рекомендацию R-7).

---

## 1. Итог (executive summary)

Архитектура зрелая: серверная ревалидация цены, идемпотентность через `unique paymentId`, HMAC-проверка вебхука, slot-mutex + `activeSlotKey unique`, fail-closed проверка слота, проверки владельца платежа. Однако найдено **2 критических** и **3 высоких** дефекта, в основном связанных с ситуацией «деньги списаны — записи нет» и с возвратами при отмене не пациентом.

| ID | Severity | Краткое описание |
|----|----------|------------------|
| BUG-01 | 🔴 Critical | У карточных платежей нет reconciler — при закрытии браузера + сбое вебхука деньги списаны, запись не создаётся, ретраев нет |
| BUG-02 | 🔴 Critical | Отмена оплаченной записи врачом/админом не делает возврат (refund только в ветке пациента) |
| BUG-03 | 🟠 High | Гонка «вебхук vs redirect»: запись создал вебхук → `epay-confirm` отдаёт 409 → пользователю показывается экран ОШИБКИ, хотя запись есть |
| BUG-04 | 🟠 High | Проверка суммы для QR fail-open: если шлюз не вернул `amount`, проверка молча пропускается (у карты — fail-closed) |
| BUG-05 | 🟠 High | «Оплачено, но записи нет» при конфликте слота или смене цены врача в процессе оплаты — без авто-возврата |
| BUG-06 | 🟡 Medium | Pending-записи и брошенные payment-intent держат слот; авто-истечения pending-записей нет |
| BUG-07 | 🟡 Medium | Сбой возврата только логируется: запись `cancelled`, но `paymentStatus='paid'`, оператор не уведомляется |
| BUG-08 | 🟡 Low-Med | Отмена < 24 ч молча сжигает оплату, без предупреждения/подтверждения в UI и без частичного возврата |
| BUG-09 | 🟢 Low | Rate-limit по IP в памяти: ложные срабатывания за NAT, сброс при рестарте |
| BUG-10 | 🟢 Low | Залогиненный пользователь может «забивать» слоты резервациями (мелкий griefing) |

## 2. Дефекты (с доказательствами)

### 🔴 BUG-01 — Карточные платежи без reconciler (потеря записи при списании)
**Где:** `signaling-server/server.js:601-627` (`reconcilePendingQRPayments` обрабатывает только `provider=halyk_qr`).
**Сценарий:** карта оплачена → пользователь закрыл вкладку до redirect на `/payment/success` → `epay-confirm` не вызван. Остаётся только вебхук `epay-callback`. Вебхук отвечает `200` **до** создания записи (`server.js:1330`), поэтому при транзиентном сбое Strapi внутри обработчика ePay **не повторит** доставку, а никакого фонового сверщика для карт нет.
**Последствие:** деньги списаны, запись не создана, ничего не ретраит. У QR такой риск закрыт сверщиком, у карты — нет.
**Фикс:** добавить периодический reconciler для `provider=epay_card` со статусом `pending`/`paid` (по аналогии с QR), сверяющий транзакцию через `EPAY_TRANSACTION_STATUS_URL` и до-создающий запись идемпотентно.

### 🔴 BUG-02 — Отмена оплаченной записи врачом/админом не возвращает деньги
**Где:** `appointment.ts:656-677` и `:719-748` — `shouldRequestRefund`/`requestAutomaticRefund` находятся **только** внутри ветки `if (isPatient)`. Ветка врача (`:686-696`) и ветка админа (`:609-632`) меняют `statuse` на `cancelled` без какого-либо возврата.
**Сценарий:** врач (или админ) отменяет оплаченный приём → пациенту деньги не возвращаются, уведомление об отмене уходит, но возврата нет.
**Последствие:** прямой финансовый/репутационный риск, частая реальная ситуация (врач заболел/перенёс).
**Фикс:** выносить логику возврата в общий код, срабатывающий при любом переходе в `cancelled` для `paymentStatus='paid'`; при отмене со стороны врача/клиники — возврат 100% независимо от 24-часового правила.

### 🟠 BUG-03 — Успешная запись показывается как ошибка (гонка вебхук vs redirect)
**Где:** `PaymentSuccess.jsx:63-84` (успех только при `response.ok && result.success`) против `server.js:1144-1162` (`epay-confirm` отдаёт **409** «already processed», если запись уже создана вебхуком/идемпотентностью).
**Сценарий:** вебхук пришёл первым и создал запись → пользователь редиректится на success → `epay-confirm` видит существующую запись → 409 → фронт считает это ошибкой, ретраит 3 раза, затем показывает красный экран «Ошибка, напишите в поддержку», хотя запись реально создана.
**Последствие:** false-negative, пациент паникует/пишет в саппорт по успешной брони.
**Фикс:** на фронте трактовать 409 (idempotent/already-processed) как успех — подтянуть запись по `paymentId`/`roomId` и показать success.

### 🟠 BUG-04 — Проверка суммы QR fail-open при отсутствии `amount`
**Где:** `server.js:204-212` `assertPaidAmountMatches` — `if (amount === undefined || amount === null) return` (молча пропускает). Используется в `halyk-qr-status` (`:988`) и в reconciler (`:412`). Карточный путь, наоборот, fail-closed: `Math.round(Number(undefined))=NaN !== expected` → отказ (`server.js:1222-1226`).
**Сценарий:** если ePay в статусе QR не вернул поле суммы — проверка соответствия суммы не выполняется, и оплата на неверную сумму будет принята как валидная.
**Последствие:** теоретическая возможность недоплаты по QR; несогласованность поведения карта/QR.
**Фикс:** для оплаченного статуса (`PAID`) требовать наличие суммы и совпадение; отсутствие суммы трактовать как «не верифицировано» (fail-closed), как у карты.

### 🟠 BUG-05 — «Оплачено, но записи нет» (конфликт слота / смена цены в процессе)
**Где:** создание идёт через контроллер `appointment.ts:509-574` (slot-mutex + проверка конфликта) и валидацию цены `:469-473`.
**Сценарии:**
- **TOCTOU двух QR/карт на один слот:** два `create-halyk-qr`/`epay-token` почти одновременно проходят `assertPaymentSlotAvailable` до персиста intent → оба оплачиваются → второе создание записи получает конфликт. QR при этом уходит в бесконечный `PAID_PENDING_APPOINTMENT`, затем `appointment_error` через 90 c (`BookingModal.jsx:755-761`); карта — экран ошибки. **Авто-возврата нет.**
- **Смена цены врача между token и confirm:** `epay-confirm` создаёт запись с `price = intent.amount` (старая), но контроллер сверяет с **текущей** ценой врача (`appointment.ts:469-473`) → `Invalid appointment price` → запись не создаётся при уже списанных деньгах.
**Последствие:** деньги списаны, записи нет, ручной разбор саппортом, без автоматического возврата.
**Фикс:** при невозможности создать оплаченную запись (конфликт/цена) — инициировать автоматический возврат и явно сообщать пользователю; для paid-gateway-create не пере-сверять цену с текущей (доверять зафиксированной в intent).

### 🟡 BUG-06 — Слоты «утекают» из-за pending-записей и брошенных intent
**Где:** `appointment.ts:111-119` `paymentIntentHoldFilter` держит `pending` intent 30 мин; `findBookedSlots`/`findSlotConflicts` считают занятыми все не-`cancelled` записи (`:806-814`, `:895-906`).
**Сценарий:** брошенная оплата держит слот 30 мин для всех; pending-запись (Kaspi/тест/админ-созданная), которую не оплатили/не отменили, держит слот **бессрочно** — авто-экспирации pending-записей нет.
**Последствие:** ложная недоступность слотов, деградация конверсии.
**Фикс:** фоновая джоба отмены «протухших» pending-записей; уменьшить hold или освобождать слот сразу при `failed/expired` intent. (Митигатор: Kaspi в проде `disabled: true`, `BookingModal.jsx:71`.)

### 🟡 BUG-07 — Сбой возврата только логируется
**Где:** `appointment.ts:738-747` — при ошибке `requestAutomaticRefund` запись уже `cancelled`, а `paymentStatus` остаётся `paid`; только `strapi.log.error`. Intent помечается `refund_failed` (`server.js:1496-1508`), но оператор не уведомляется.
**Последствие:** деньги не возвращены, нет сигнала оператору, расхождение состояния.
**Фикс:** уведомление админу/алерт при `refund_failed` + отчёт/дашборд по таким записям; ретрай возвратов.

### 🟡 BUG-08 — Отмена < 24 ч молча сжигает оплату
**Где:** `appointment.ts:654-676` — возврат только при `hoursUntil >= 24`. UI не предупреждает о невозврате и не запрашивает подтверждение, частичного возврата нет.
**Фикс:** показывать политику возврата и подтверждение перед отменой; рассмотреть частичный возврат.

### 🟢 BUG-09 — Rate-limit по IP в памяти
**Где:** `server.js:459-479`, применяется в `epay-token`/`create-halyk-qr`. За корпоративным NAT/прокси возможны ложные 429; сбрасывается при рестарте; распределённо не работает.
**Фикс:** ключ по userId + IP; вынести в Redis при горизонтальном масштабировании.

### 🟢 BUG-10 — Griefing резерваций слотов
**Где:** `server.js:1923-1963` `reserve-slot` — залогиненный пользователь может зарезервировать много слотов (TTL 5 мин каждый). Сокеты требуют аутентификации (`server.js:1666-1676`), так что аноним исключён, но злоупотребление возможно.
**Фикс:** лимит активных резерваций на пользователя.

**Наблюдение (не баг):** жёсткий офсет Казахстана `+05:00` (`server.js:1067`, `appointment.ts:404,417`). Сейчас корректно (в РК нет переходов, единый UTC+5), но хрупко при будущих изменениях TZ-политики.

## 3. Что сделано хорошо (подтверждено)
- Серверная ревалидация цены против `doctor.price` на всех путях (`epay-token:1551`, `create-halyk-qr:815`, `appointment create:469`).
- Идемпотентность: `appointment.paymentId` и поля intent `unique`; множественные пути (status-poll, webhook, reconciler) сходятся к одной записи.
- HMAC-SHA512 проверка вебхука с `timingSafeEqual` (`server.js:1316-1327`).
- Проверка владельца платежа: `accountId`/patientId/doctor/dateTime в `epay-confirm` (`server.js:1228-1249`), ownership в QR-status (`:943-945`).
- Конкурентность: slot-mutex (`appointment.ts:184-202`) + `activeSlotKey unique` как защита на уровне БД.
- Fail-closed проверка слота при недоступности Strapi (`server.js:1100-1104`, `assertPaymentSlotAvailable:526-543`).
- Грейс на создание оплаченной записи у границы слота (`appointment.ts:374-383`).
