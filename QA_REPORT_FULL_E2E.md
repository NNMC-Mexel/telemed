## 3. Находки (с доказательствами)

### 🟠 F-01 — Врач может пометить будущий приём `completed` без проверки времени
**Где:** [appointment.ts](server/src/api/appointment/controllers/appointment.ts) — ветка врача в `update()`: `DOCTOR_ALLOWED_STATUSES = ['confirmed','in_progress','completed','cancelled']`, статус ставится без проверки `now >= dateTime`.
**Последствие:**
- `review.create` ([review.ts](server/src/api/review/controllers/review.ts)) пускает отзыв только при `statuse==='completed'` → врач может «открыть» отзыв на не состоявшийся приём.
- Окно загрузки медзаключений завязано на конец консультации (это `medical-document.ts` проверяет отдельно — там есть guard по времени, так что документы защищены), но статус-целостность нарушается.
**Фикс:** запрещать переход в `completed`, пока `now < dateTime + slotDuration` (или хотя бы `now < dateTime`); опционально — только из `in_progress`.

### 🟠 F-02 — Состояние в памяти не переживает горизонтальное масштабирование
**Где:** [server.js](signaling-server/server.js) — `pendingSlotReservations`, `rateLimitStore`, `confirmedInvoices`, `pendingCardPayments`, `pendingQRPayments`, `tokenCache`, `roomAccessCache`; плюс in-memory rate-limit в [rate-limit.ts](server/src/middlewares/rate-limit.ts) и [check-email.ts](server/src/api/check-email/controllers/check-email.ts).
**Последствие:** при >1 инстанса signaling/Strapi: резервации слотов и rate-limit не видны другим репликам, `confirmedInvoices`/`pendingCardPayments` расходятся. Идемпотентность платежей при этом защищена БД (`unique paymentId`), так что **двойных списаний не будет**, но возможны: гонки резерваций между репликами, неравномерный rate-limit, лишние ретраи.
**Фикс:** вынести разделяемое состояние в Redis (резервации, rate-limit, idempotency-locks) перед масштабированием; либо «липкие» сессии + единственный инстанс signaling.

### 🟠 F-03 — `pending` записи держат слот бессрочно
**Где:** [appointment.ts](server/src/api/appointment/controllers/appointment.ts) `findBookedSlots`/`findSlotConflicts` — занятыми считаются все не-`cancelled` записи (`statuse $ne cancelled` / `$in active`). Запись в статусе `pending` (создаётся в test-режиме, через Kaspi, или админом) не имеет авто-экспирации.
**Последствие:** неоплаченная/«зависшая» запись блокирует время навсегда. (Kaspi в проде `disabled`, но путь `pending` остаётся для test/admin.) Перекликается с уже исправленным BUG-06 (там закрыта intent-сторона).
**Фикс:** фоновая джоба, переводящая «протухшие» `pending` (старше N часов и без оплаты) в `cancelled`/`failed`.

### 🟠 F-04 — Жёстко зашитый офсет UTC+5
**Где:** `appointment.ts:404,417` (рабочие часы/дни), `server.js` slot-verify (`+05:00`), `findBookedSlots`/`findSlotConflicts` (`KZ_OFFSET_MS`).
**Последствие:** сегодня корректно (РК — единый UTC+5, без переходов), но при будущих изменениях TZ-политики или мульти-региональности всё «поедет» молча.
**Фикс:** вынести смещение в конфиг/таймзону (`Asia/Almaty`) через библиотеку TZ; не хардкодить минуты.

### 🟡 F-05 — Енумерация email
**Где:** [check-email.ts](server/src/api/check-email/controllers/check-email.ts) возвращает `{ exists: true|false }` (с rate-limit и фиксированной задержкой), тогда как `forgotPassword`/`sendEmailConfirmation` ([strapi-server.ts](server/src/extensions/users-permissions/strapi-server.ts)) намеренно enumeration-safe (`ok/sent` для любых).
**Последствие:** непоследовательная политика приватности — endpoint позволяет проверять наличие аккаунта по email.
**Фикс (по продукту):** если проверка нужна для UX регистрации — оставить, но усилить (CAPTCHA/жёстче лимит); иначе убрать прямой ответ существования.

### 🟡 F-06 — `medical-document.update()` без allowlist полей для пациента
**Где:** [medical-document.ts](server/src/api/medical-document/controllers/medical-document.ts) `update()` — ветка проверок только `if (!isAdmin && isDoctor)`. Для пациента (владельца, прошедшего policy) выполняется `update({ data: body })` с произвольными полями.
**Последствие:** пациент на **своём** документе может переписать `sharedWithDoctors` напрямую (минуя валидацию реальных врачей в `share()`), перепривязать `appointment`/`doctor`, отдать `user`. Данные свои, утечки чужого нет, но обходятся задуманные контроли (например, расшарить документ любому doctorId).
**Фикс:** для пациента в `update()` применять allowlist (`title`, `description`, `type`, `file`); изменение sharing — только через `share()`.

### 🟡 F-07 — Неограниченные клиентские лимиты
**Где:** [message.ts](server/src/api/message/controllers/message.ts) `find` — `limit: Number(query.pagination.limit || ... || 500)` без верхней границы; [server.js](signaling-server/server.js) `chat-message` — нет лимита длины строки.
**Последствие:** запрос больших выборок в рамках своих бесед; гигантское сообщение хранится в памяти комнаты и рассылается. Мелкий DoS.
**Фикс:** ограничить `limit` (например, ≤500) и длину `content` (например, ≤4–8 KB).

### 🟡 F-08 — Auth rate-limit в памяти по IP
**Где:** [rate-limit.ts](server/src/middlewares/rate-limit.ts), [check-email.ts](server/src/api/check-email/controllers/check-email.ts) — тот же класс, что платёжный BUG-09 (там уже улучшено: мягкий IP + строгий per-user). На auth-эндпоинтах ключ только по IP.
**Последствие:** за общим NAT легитимные пользователи мешают друг другу; сброс при рестарте; распределённо не работает (см. F-02).
**Фикс:** Redis-бэкенд + (где возможно) комбинированный ключ; для login оставить IP, но повысить порог и логировать.

### 🟢 F-09 — `unreadCount`/`markAllAsRead` ограничены 1000
**Где:** [notification.ts](server/src/api/notification/controllers/notification.ts) — оба используют `limit: 1000`.
**Последствие:** у пользователя с >1000 непрочитанных счётчик занижен и «прочитать всё» оставит часть. Маловероятно, но неверно.
**Фикс:** считать через `count()`; помечать пачками в цикле до нуля.

### ⚪ F-10 — Мелкие несоответствия
- `statuse: 'no_show'` есть в схеме, но воркфлоу его нигде не выставляет (нет автоматики неявки).
- Отзыв: `appointment.rating` валиден 1–10, UI обычно 1–5 — прямой POST с rating=10 пройдёт. Уточнить шкалу.

## 4. Что проверено и сделано надёжно ✅

- **Ownership везде дважды:** route-policies (`is-owner`, `is-appointment-participant`, `is-medical-document-participant`, `is-conversation-member`, `is-doctor-profile-owner`, `is-timeslot-owner`) **и** фильтрация в контроллерах (find/findOne/update/delete). IDOR закрыт фильтрацией строго по `documentId`.
- **Регистрация (B2B):** публично только пациент; `userRole=doctor|admin` и `doctorData` → `403`; обязательные согласия с версионированием; подтверждение email с TTL 24 ч; нормализация и уникальность телефона/ИИН; **rollback** при сбое.
- **Консультация:** `join-room` требует членства (пациент/врач приёма) **и** живой проверки окна `can-join` (серверное время; будущие/отменённые/завершённые не пускаются); WebRTC-релей (`offer/answer/ice`) только между пирами одной комнаты; идентичность в чате берётся из серверного состояния; `force-end-call` — только врач.
- **Доктор-профиль:** non-admin не может менять `price/rating/isActive/licenseNumber/...`; активный врач обязан иметь лицензию (lifecycle).
- **Файлы:** `file-proxy` отдаёт медфайлы только владельцу/врачу/расшаренному/админу; публичны лишь фото активных врачей с безопасным MIME; приватные — `no-store` + `attachment`.
- **Отзывы:** только завершённый приём, принадлежащий пациенту, у соответствующего врача, один на приём.
- **Платежи/возвраты:** идемпотентность через БД, HMAC-вебхук, авто-возвраты, эскалация админам — закрыто в прошлом цикле.

## 6. Рекомендации (приоритет)
1. **R-1 (F-01):** guard на переход в `completed` (по времени/из `in_progress`).
2. **R-2 (F-03):** джоба авто-экспирации `pending`-записей.
3. **R-3 (F-02/F-08):** перенос разделяемого состояния и rate-limit в Redis до масштабирования.
4. **R-4 (F-04):** перейти на именованную таймзону `Asia/Almaty`.
5. **R-5 (F-06):** allowlist полей в `medical-document.update()` для пациента.
6. **R-6 (F-07/F-09):** ограничить `limit`/длину сообщений; счётчики через `count()`.
7. **R-7 (F-05):** пересмотреть политику `check-email` (CAPTCHA или убрать прямой ответ).
