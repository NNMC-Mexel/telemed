# QA-отчёт: полное тестирование телемедицинской системы MedConnect

**Дата:** 2026-06-11
**QA-инженер:** Claude (Fable 5)
**Окружение:** локальное — Strapi v5 backend (`http://localhost:1340`), React+Vite frontend, SQLite dev-БД
**Метод:** анализ кода (политики доступа, контроллеры, lifecycle-хуки, фронтенд-гарды) + живое тестирование REST API под тремя ролями (Пациент / Доктор / Админ), включая негативные и IDOR-сценарии.

> **Важно про окружение.** Тестирование велось против **dev-конфигурации**: `PAYMENTS_LIVE=false`, `RATE_LIMIT_ENABLED=false`, `NODE_ENV=development`, без подключённого MINIO/S3 (file-proxy отвечал «Storage not configured»). Часть защит (живые платежи, rate-limit, реальная отдача файлов) в этом режиме отключена сознательно — их поведение оценено по коду, а не вживую. Это отмечено в каждом релевантном кейсе.

## 2. Список критических багов

**Critical (блокеры релиза): не обнаружено.** Ядро RBAC/IDOR/конкурентности устойчиво.

### High

**BUG-01 — Нет серверной политики паролей (A-07).**
- **Где:** `server/src/extensions/users-permissions/strapi-server.ts`, `register`.
- **Суть:** прямой вызов `POST /auth/local/register` с `password:"123"` **успешно создаёт пользователя**. Клиентская валидация требует ≥6 символов, но сервер её не дублирует. Любой может зарегистрироваться со слабым/коротким паролем в обход UI.
- **Риск:** компрометация аккаунтов с медицинскими данными через брутфорс/словарь.
- **Рекомендация:** добавить на бэкенде проверку минимальной длины (рекомендуется ≥8–12), сложности и проверку по списку скомпрометированных паролей до вызова `originalRegister`. Также применить к смене пароля и reset-password.

1. **UX-01 (D-10): Лочинг редактирования профиля доктора без лицензии.** Если у активного доктора не задан `licenseNumber`, **любое** сохранение профиля падает с `400 licenseNumber is required for active doctors` — доктор не может отредактировать даже bio. Нужно: либо явное и понятное сообщение в UI с указанием, что лицензию задаёт админ, либо разрешить правку неконфликтующих полей.
2. **UX-02 (P-22): In-memory mutex от двойного бронирования — per-process.** При горизонтальном масштабировании backend (>1 инстанс) мьютекс `slotLocks` не защитит от гонки между инстансами; останется только пост-проверка существующих записей (есть, но без БД-уникального ограничения возможна узкая гонка). Рекомендуется уникальный индекс на `activeSlotKey` в БД как финальная страховка.
3. **UX-03: Сообщения об ошибках — смешанный язык.** API отдаёт технические сообщения на английском (`"dateTime must be in the future"`, `"Invalid appointment price"`), при этом часть — на русском (SLOT_CONFLICT). Для конечного пользователя фронт должен маппить коды в локализованные тексты; нужно убедиться, что все ошибки 400 имеют человекочитаемый перевод.
4. **UX-04: Регистрация многошаговая (validateStep1/Step2/DoctorStep).** В коде формы остались ветки для doctorStep, хотя публичная регистрация докторов запрещена (мёртвый путь) — убрать, чтобы не путать.
5. **UX-05: Поведение при пустых списках / долгой загрузке / ошибке сервера** — не проверено вживую (нужен прогон UI). Заявлен `LoadingScreen`, но состояния «пусто»/«ошибка сети»/«сервер 500» на экранах списков требуют ручной проверки.

## 4. Список проблем с безопасностью

| ID | Severity | Проблема | Рекомендация |
|---|---|---|---|
| BUG-01 | High | Нет серверной валидации пароля | Ввести политику паролей на бэкенде (длина/сложность/blocklist) |
| BUG-02 | Medium | Создание `paymentStatus:paid` пациентом разрешено в test-режиме (P-16) | Гарантировать `PAYMENTS_LIVE=true` в проде; добавить старт-чек, падающий если в production `PAYMENTS_LIVE!=true`. В коде guard есть только для `NODE_ENV=production && !isPaymentsLive` — убедиться, что прод всегда стартует с этим флагом |
| BUG-03 | Medium | JWT в `localStorage` (S-03) | Рассмотреть httpOnly+SameSite cookie; как минимум — строгая CSP (есть) + короткий TTL токена + ротация |
| BUG-04 | Medium | rate-limit отключён в QA-конфиге (S-04) | Проверить вживую, что в проде `RATE_LIMIT_ENABLED=true` и лимиты на login/register/forgot реально срабатывают |
| BUG-08 | Low | `/auth/check-email` раскрывает существование email (A-17) | Либо убрать эндпоинт, либо вернуть единый ответ; несогласованно с enumeration-safe forgot/resend |
| SEC-info | Low | Секреты SMTP лежат в `server/.env` (в .gitignore — не в репозитории, проверено) | OK. Перед продом ротировать тестовый SMTP-пароль, который сейчас в dev `.env` |

**Положительные находки по безопасности (подтверждено тестами):**
- IDOR на appointments/notifications/doctors/medical-documents — **закрыт** (403 на всех попытках).
- Эскалация роли через регистрацию — **невозможна** (403 на doctor/admin).
- Подделка цены и накрутка рейтинга/отзывов доктором — **отклоняется**.
- `/users/me` и admin-выдача users — **не содержат** password/токенов сброса.
- file-proxy — манипуляция доступом к PHI требует JWT и проверки владения; публичны только image-аватары докторов.
- Гонка двойного бронирования — **исключена** (ровно 1 запись).
- can-join видео — окно считается по **серверному** времени (защита от подмены часов).

## 5. Список недостающих тест-кейсов (рекомендуется добавить)

1. **Каскадное удаление пользователя** (AD-07): что с `appointments`, `medical-documents`, `conversations`, `reviews`, `consent-records` при hard-delete пациента/доктора. Проверить отсутствие orphan-записей и сохранность истории/аудита.
2. **Live-платежи end-to-end** (`PAYMENTS_LIVE=true`): успех/неуспех ePay, идемпотентность по `paymentId`, авто-refund при SLOT_CONFLICT (BUG-05 в коде), refund при отмене доктором/админом, уведомление админов при сбое refund (BUG-07).
3. **Rate-limit вживую**: brute-force login/register/forgot при `RATE_LIMIT_ENABLED=true`.
4. **file-proxy с реальным MINIO/S3**: отдача PHI авторизованному владельцу/доктору, 403 чужому, корректные заголовки `attachment/no-store`.
5. **Чат/WebRTC**: отправка сообщений только участником conversation, подключение к видео в окне/вне окна, поведение при потере сети, права на присоединение к чужой комнате.
6. **UI/UX вживую (desktop/tablet/mobile + Capacitor)**: адаптивность, длинные имена/email/телефоны, пустые списки, спиннеры долгой загрузки, экран ошибки сервера, истёкшая сессия (редирект на login через 401-интерсептор).
7. **Прямой доступ по URL во фронте** (`/admin`, `/doctor`, `/patient` под чужой ролью) — `ProtectedRoute` редиректит по `userRole`, но нужен ручной прогон, в т.ч. при ручной правке `localStorage`.
8. **Часовые пояса**: записи на границе суток (19:00 UTC = 00:00 KZ+1) — корректность booked-slots и отображения дат у пользователей в разных TZ.
9. **Полнота локализации**: пословная проверка ru/kk/en на всех экранах, fallback, отсутствие «голых» ключей.
10. **Перенос записи (reschedule)**: в API явного reschedule нет (только cancel + новая запись) — подтвердить продуктовое решение или добавить кейс.
11. **Загрузка файлов пациентом**: лимиты размера/типа, антивирус/санитизация, превью, отзыв доступа (sharedWithDoctors).
12. **Идемпотентность двойного клика на отмене/подтверждении** (не только на создании).

## 6. Рекомендации перед релизом

**Обязательно (go/no-go):**
1. **Закрыть BUG-01** — добавить серверную политику паролей. Это High и единственный явный функциональный дефект безопасности.
2. **Гарантировать прод-конфиг:** `PAYMENTS_LIVE=true`, `RATE_LIMIT_ENABLED=true`, `NODE_ENV=production`. Рекомендуется добавить fail-fast проверку при старте сервера, которая не даёт подняться проду с небезопасными значениями (BUG-02, BUG-04).
3. **Прогнать недостающие кейсы №1, №2, №3, №4** (каскадное удаление, live-платежи/refund, rate-limit, file-proxy на реальном хранилище) — это зоны, не покрытые в dev.

**Желательно:**
4. Перевести хранение сессии на httpOnly-cookie или сократить TTL JWT + ротация (BUG-03).
5. Привести `/auth/check-email` к enumeration-safe поведению или удалить (BUG-08).
6. Добавить уникальный БД-индекс на активный слот (`activeSlotKey`) как страховку при мультиинстансном backend (UX-02).
7. UX: понятное сообщение/флоу для доктора без лицензии (UX-01), убрать мёртвый doctor-step в регистрации (UX-04), проверить локализацию всех ошибок (UX-03).
8. Ротировать тестовые секреты (SMTP) перед продом.

**Сильные стороны (готово к проду):** разграничение прав по ролям, защита от IDOR, контроль цены/платёжного статуса/рейтинга, защита от двойного бронирования, серверная проверка времени видео-окна, санитизация выдачи пользователей, CORS/CSP/HSTS, enumeration-safe сброс пароля, блокировка неподтверждённых и заблокированных аккаунтов.
